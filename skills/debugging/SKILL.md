# Debugging Skill

系統化的除錯方法論，高效定位和解決問題。

## 除錯流程

### Phase 1: 情報收集 🔍

1. **重現問題**
   - 確定能穩定重現問題的步驟
   - 記錄環境資訊（OS、版本、設定）
   
2. **收集錯誤資訊**
   - 完整的錯誤訊息和 stack trace
   - 相關的 log 輸出
   - 網路請求/回應（如適用）

3. **定義預期 vs 實際行為**
   - 預期：應該發生什麼
   - 實際：實際發生什麼

### Phase 2: 假設建立 💡

根據收集的資訊，列出可能的原因（從最可能到最不可能）：

```markdown
## 可能原因分析

1. **[原因 A]** - 可能性：高
   - 支持證據：[...]
   - 如何驗證：[...]

2. **[原因 B]** - 可能性：中
   - 支持證據：[...]
   - 如何驗證：[...]
```

### Phase 3: 假設驗證 🧪

對每個假設進行驗證：

1. **二分法縮小範圍**
   - 註解掉一半代碼，看問題是否還存在
   - 逐步縮小問題範圍

2. **最小可重現案例**
   - 移除所有無關代碼
   - 建立最小的問題重現範例

3. **對照測試**
   - 在不同環境/條件下測試
   - 確認問題的觸發條件

### Phase 4: 修復與驗證 ✅

1. **實施修復**
   - 修改代碼
   - 確保修復不引入新問題

2. **回歸測試**
   - 原問題已解決
   - 相關功能正常運作
   - 撰寫測試防止問題復發

3. **記錄學習**
   - 問題根因是什麼
   - 如何避免類似問題

## 常見問題類型速查

### 🔴 Runtime Errors
```
檢查項目：
□ 變數是否已初始化
□ 型別是否正確
□ 空值/undefined 處理
□ 陣列越界
□ 非同步操作順序
```

### 🟠 Logic Errors
```
檢查項目：
□ 條件判斷邏輯
□ 迴圈終止條件
□ 邊界條件處理
□ 狀態更新時機
```

### 🟡 Performance Issues
```
檢查項目：
□ N+1 查詢問題
□ 不必要的重新渲染
□ 記憶體洩漏
□ 無效的快取
□ 過大的資料處理
```

### 🔵 Integration Issues
```
檢查項目：
□ API 合約是否匹配
□ 資料格式轉換
□ 認證/授權
□ 網路/超時設定
□ 版本相容性
```

## Debug 輸出格式

```markdown
## 🐛 Debug Report

### 問題描述
[簡述問題]

### 重現步驟
1. [步驟 1]
2. [步驟 2]

### 根因分析
[問題的根本原因]

### 解決方案
[修復方式和代碼變更]

### 預防措施
[如何避免類似問題]
```
